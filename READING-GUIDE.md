# 閱讀指南：開始之前

## 這是什麼

這份研究材料將一個開源專案的源碼拆解為多個漸進式版本，從最小可行核心逐步增加複雜度。

**它不是**：API 文件、使用教學、或逐行程式碼註解。

**它是**：還原設計者的思考過程——面對什麼問題、為什麼選擇這個方案、放棄了什麼替代方案。

## 每一版的閱讀重點

每個版本都從一個**具體問題**開始。在往下讀之前，先問自己：

> 「如果是我遇到這個問題，我會怎麼解？」

帶著你自己的想法再往下看，才能理解設計者的選擇為什麼合理（或者你會發現一個更好的解法——那也很好）。

每一版的結構：

| 段落 | 你的關注點 |
|------|----------|
| **設計故事** | 最重要。理解問題本身。如果你不覺得這個問題真的是問題，後面的程式碼就沒有意義 |
| **核心概念** | 理解解法的心智模型。圖表比文字重要——能在腦中重建這張圖，就算讀懂了 |
| **注意事項** | 知道這個設計的邊界和取捨。沒有完美設計，只有適合場景的設計 |
| **程式碼實作** | 驗證你的理解。如果你已經從心智模型推導出大致的程式碼結構，這裡只是確認 |
| **使用範例** | 確認你能預測行為。看範例時先遮住結果，自己猜答案 |
| **與源碼對照** | 連結到真實源碼。有興趣時再深入，不需要每次都看 |

## 怎麼知道你讀懂了

每讀完一版，試著用一句話回答：

> 「這個版本解決了什麼問題？如果沒有它，會發生什麼？」

如果你能回答這兩個問題，就可以進入下一版。如果答不出來，重讀「設計故事」。

## 不需要順序讀

版本之間是漸進式的——後面的版本建立在前面的基礎上。但如果你對某個主題已經有概念，可以直接跳到那個版本。每個版本的設計故事都會交代足夠的前情。
